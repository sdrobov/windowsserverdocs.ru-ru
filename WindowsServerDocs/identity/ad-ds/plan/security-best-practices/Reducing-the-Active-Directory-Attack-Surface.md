# <a name="reducing-the-active-directory-attack-surface"></a>Снижение уязвимостей Active Directory

>Область применения: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012

В этом разделе рассматриваются технические элементы управления, которые необходимо реализовать для уменьшения уязвимой зоны установки Active Directory. Раздел содержит следующие сведения:  
  
-   [Реализация административных моделей с минимальными правами доступа](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Implementing-Least-Privilege-Administrative-Models.md) посвящена определению риска использования учетных записей с высоким уровнем привилегий для повседневного администрирования, а также рекомендации по реализации для снижения риска наличия привилегированных учетных записей.  
  
-   [Реализация защищенных административных узлов](../../../ad-ds/plan/security-best-practices/Implementing-Secure-Administrative-Hosts.md) описывает принципы развертывания выделенных и безопасных административных систем в дополнение к некоторым примерам подходов к безопасному развертыванию административного узла.  
  
-   [Защита контроллеров домена от атак](../../../ad-ds/plan/security-best-practices/Securing-Domain-Controllers-Against-Attack.md) обсуждаются политики и параметры, которые, несмотря на рекомендации по реализации защищенных административных узлов, содержат некоторые рекомендации, связанные с контроллером домена, которые гарантируют, что контроллеры домена и системы, используемые для управления ими, надежно защищены.  
  
## <a name="privileged-accounts-and-groups-in-active-directory"></a>Привилегированные учетные записи и группы в Active Directory  
В этом разделе содержатся общие сведения о привилегированных учетных записях и группах в Active Directory, предназначенных для объяснения общие черты и различий между привилегированными учетными записями и группами в Active Directory. Понимание этих различий заключается в том, что вы реализуете рекомендации в области [реализации административных моделей с минимальными привилегиями](../../../ad-ds/plan/security-best-practices/../../../ad-ds/plan/security-best-practices/Implementing-Least-Privilege-Administrative-Models.md) или настраиваете их для своей организации, у вас есть средства, необходимые для защиты каждой группы и учетной записи соответствующим образом.  
  
### <a name="built-in-privileged-accounts-and-groups"></a>Встроенные привилегированные учетные записи и группы  
Active Directory упрощает делегирование администрирования и поддерживает принцип наименьших привилегий в назначении прав и разрешений. Обычные пользователи, имеющие учетные записи в домене, по умолчанию могут читать значительную часть данных, хранящихся в каталоге, но могут изменять только очень ограниченный набор информации в каталоге. Пользователям, которым требуется дополнительная привилегия, можно предоставить членство в различных "привилегированных" группах, встроенных в каталог, чтобы они могли выполнять определенные задачи, связанные с их ролями, но не могут выполнять задачи, не относящиеся к их обязанностям. Организации также могут создавать группы, адаптированные к конкретным должностным обязанностям, и получают детализированные права и разрешения, позволяющие ИТ-специалистам выполнять повседневные административные функции без предоставления прав и разрешений, превышающих для этих функций требуется.  
  
В Active Directory три встроенных группы являются самыми высокими группами прав в каталоге: Администраторы предприятия, Администраторы домена и Администраторы. Конфигурация и возможности по умолчанию для каждой из этих групп описаны в следующих разделах:  
  
#### <a name="highest-privilege-groups-in-active-directory"></a>Наибольшие группы прав в Active Directory  
  
##### <a name="enterprise-admins"></a>Администраторы предприятия  
Администраторы предприятия (EA) — это группа, которая существует только в корневом домене леса, и по умолчанию она является членом группы администраторов во всех доменах леса. Встроенная учетная запись администратора в корневом домене леса является единственным элементом по умолчанию группы EA. EAs предоставляет права и разрешения, позволяющие им реализовывать изменения на уровне леса (то есть изменения, затрагивающие все домены в лесу), такие как добавление или удаление доменов, установка доверий лесов или повышение функциональных уровней леса. В правильно спроектированной и реализованной модели делегирования членство в EA требуется только при первом создании леса или при внесении определенных изменений на уровне леса, например при установлении отношения доверия с исходящим лесом. Большинство прав и разрешений, предоставляемых группе EA, могут быть делегированы пользователям и группам с меньшими привилегиями.  
  
##### <a name="domain-admins"></a>Администраторы домена  

Каждый домен в лесу имеет собственную группу Администраторы домена (DA), которая является членом группы администраторов этого домена, и членом локальной группы администраторов на каждом компьютере, присоединенном к домену. Единственным элементом по умолчанию группы DA для домена является встроенная учетная запись администратора для этого домена. DAs — это "все мощные" в своих доменах, в то время как EAs имеет права уровня леса. В правильно спроектированной и реализованной модели делегирования членство в группе "Администраторы домена" должно быть обязательным только в сценариях "прозрачное стекло" (например, в ситуациях, когда требуется учетная запись с высоким уровнем привилегий на каждом компьютере в домене). Несмотря на то, что собственные механизмы делегирования Active Directory позволяют делегировать возможность использовать учетные записи DA только в экстренных сценариях, создание эффективной модели делегирования может занять много времени, и многие организации используют сторонние средства для ускорения процесса.  
  
##### <a name="administrators"></a>Администраторы  
Третья группа — это встроенная группа локальных администраторов домена (BA), в которую вложены DAs и EAs. Этой группе предоставляются многие прямые права и разрешения в каталоге и на контроллерах домена. Однако Группа администраторов для домена не имеет привилегий на рядовых серверах или на рабочих станциях. Это осуществляется посредством членства в группе локальных администраторов компьютеров, которой предоставлены локальные привилегии.  
  
> [!NOTE]  
> Хотя эти группы по умолчанию являются конфигурациями этих привилегированных групп, член любой из трех групп может управлять каталогом, чтобы получить членство в любой из других групп. В некоторых случаях несложно получить членство в других группах, а в других — более сложная задача, но с точки зрения потенциального права все три группы должны считаться фактически эквивалентными.  
  
##### <a name="schema-admins"></a>Администраторы схемы  

Четвертая привилегированная группа, Администраторы схемы (SA), существует только в корневом домене леса и имеет только встроенную учетную запись администратора домена в качестве члена по умолчанию, аналогично группе "Администраторы предприятия". Группа Администраторы схемы предназначена для заполнения только временно и иногда (если требуется изменение схемы AD DS).  
  
Хотя Группа SA является единственной группой, которая может изменять схему Active Directory (то есть базовые структуры данных каталога, такие как объекты и атрибуты), область прав и разрешений группы SA более ограничена, чем ранее описанная выше. группы. Также распространено, что организации разработали соответствующие методики для управления членством в группе SA, так как членство в группе обычно нередко требуется и только в течение короткого периода времени. Это технически верно для групп EA, DA и Ба в Active Directory, но гораздо менее распространено, что организации реализовали аналогичные методики для этих групп, как и для группы SA.  
  
#### <a name="protected-accounts-and-groups-in-active-directory"></a>Защищенные учетные записи и группы в Active Directory  
В Active Directory набор привилегированных учетных записей и групп по умолчанию под названием «защищенные» учетные записи и группы защищены не так, как другие объекты в каталоге. Любая учетная запись, имеющая прямое или транзитивное членство в любой защищенной группе (независимо от того, является ли членство производной от групп безопасности или распространения) наследует эту ограниченную безопасность.  

  
Например, если пользователь является членом группы рассылки, которая, в свою очередь, является членом защищенной группы в Active Directory, этот объект пользователя помечается как защищенная учетная запись. Если учетная запись помечена как защищенная, значение атрибута Админкаунт объекта задается равным 1.  
  
> [!NOTE]
> Хотя транзитивное членство в защищенной группе включает в себя вложенные и вложенные группы безопасности, учетные записи, являющиеся членами вложенных групп распространения, не будут получать SID защищенной группы в маркерах доступа. Однако группы распространения можно преобразовать в группы безопасности в Active Directory, поэтому группы рассылки включаются в перечисление защищенных членов группы. Если защищенная вложенная группа распространения постоянно преобразуется в группу безопасности, учетные записи, которые являются членами прежней группы распространения, впоследствии будут получать идентификатор безопасности родительской защищенной группы в маркерах доступа при следующем входе в систему.

##### <a name="adminsdholder-and-sdprop"></a>AdminSDHolder и SDProp  
В системном контейнере каждого Active Directory домена автоматически создается объект с именем AdminSDHolder. Объект AdminSDHolder предназначен для обеспечения согласованного применения разрешений для защищенных учетных записей и групп независимо от того, где находятся защищенные группы и учетные записи в домене.  

Каждые 60 минут (по умолчанию) процесс, известный как распространитель дескрипторов безопасности (SDProp), выполняется на контроллере домена, который содержит роль эмулятора PDC домена. SDProp сравнивает разрешения для объекта AdminSDHolder домена с разрешениями на защищенные учетные записи и группы в домене. Если разрешения для любой из защищенных учетных записей и групп не совпадают с разрешениями для объекта AdminSDHolder, то разрешения на защищенные учетные записи и группы сбрасываются в соответствии с параметрами объекта AdminSDHolder домена.  
  
Наследование разрешений отключено для защищенных групп и учетных записей. Это означает, что даже если учетные записи или группы перемещаются в разные расположения в каталоге, они не наследуют разрешения от своих новых родительских объектов. Наследование также отключено для объекта AdminSDHolder, поэтому изменения разрешений для родительского объекта не изменяют разрешения объекта AdminSDHolder.  
  
> [!NOTE]
> Если учетная запись удаляется из защищенной группы, она больше не считается защищенной учетной записью, но ее атрибут Админкаунт остается равным 1, если он не был изменен вручную. В результате этой конфигурации списки ACL объекта больше не обновляются с помощью SDProp, но объект по-прежнему не наследует разрешения от своего родительского объекта. Таким образом, объект может находиться в подразделении (OU), к которому были делегированы разрешения, но ранее защищенный объект не будет наследовать эти делегированные разрешения. Сценарий поиска и сброса ранее защищенных объектов в домене можно найти в [Служба поддержки Майкрософт статье 817433](https://support.microsoft.com/?id=817433).  
  
###### <a name="adminsdholder-ownership"></a>Владение объектом AdminSDHolder  
Большинство объектов в Active Directory принадлежат группе Ба домена. Однако объект AdminSDHolder по умолчанию принадлежит группе DA домена. (Это обстоятельство, когда DAs не наследует свои права и разрешения через членство в группе администраторов для домена.)  
  
В версиях Windows, предшествующих Windows Server 2008, владельцы объекта могут изменять разрешения объекта, включая предоставление самих разрешений, которые им изначально не приходилось. Таким образом, разрешения по умолчанию для объекта AdminSDHolder домена запрещают пользователям, которые являются членами группы или EA, изменять разрешения для объекта AdminSDHolder домена. Тем не менее члены группы администраторов домена могут становиться владельцами объекта и предоставлять себе дополнительные разрешения, что означает, что эта защита является элементарной и защищает объект от случайной модификации пользователями, которые не являются членами группы DA в домене. Кроме того, группы Ба и EA (где применимо) имеют разрешение на изменение атрибутов объекта AdminSDHolder в локальном домене (корневой домен для EA).  
  
> [!NOTE]  
> Атрибут объекта AdminSDHolder, Дшеуристикс, позволяет ограниченную настройку (удаление) групп, которые считаются защищенными группами и зависят от AdminSDHolder и SDProp. Эту настройку следует тщательно рассмотреть, если она реализована, хотя существуют допустимые обстоятельства, в которых полезно изменение Дшеуристикс в AdminSDHolder. Дополнительные сведения об изменении атрибута Дшеуристикс объекта AdminSDHolder можно найти в служба поддержки Майкрософт статьях [817433](https://support.microsoft.com/?id=817433) и [973840](https://support.microsoft.com/kb/973840), а также в [приложении C: защищенные учетные записи и группы в Active Directory](Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).

Дополнительные сведения обо всех стандартных и встроенных группах по умолчанию в Active Directory и правах пользователей, назначенных каждому, см. [в приложении б. привилегированные учетные записи и группы в Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-B--Privileged-Accounts-and-Groups-in-Active-Directory.md).
